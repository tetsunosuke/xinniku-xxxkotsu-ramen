## 🎭 ストーリー・シナリオの問題点

### S-01｜パスコードのロジックが矛盾している

**重大度：高**

- `CORRECT_PASSCODE = '19940826'` と実装されている（生まれた西暦 `1994` + 店のオープン日 `0826`）
- しかし太郎のSNSプロフィールには「🎂 6月13日生まれ ♊」と記載されており、**生年は書かれていない**
- 店のオープン日は `since 2019.08.26` とブラウザのHPに記載されているが、**太郎のSNSには明示的な「オープン日」記述がない**（周年記念ポストの日付から類推する必要がある）
- 太郎のパスワードルールのヒント投稿に「自分が生まれた年の西暦を先頭に」とあるが、**生年（1994年）を知る方法がゲーム内にない**。ケンのSNSに「今日で32歳」（2026年3月5日）という投稿があり「同い年」ということは分かるが、計算が必要なうえ**ケンの誕生日しか直接提示されていない**
- **改善案：** 太郎のプロフィール文またはケンの投稿に「同い年の1994年生まれ」と明示するか、プロフィールの「2018年から利用」付近に生年が確認できる情報を追加する

→ これはケンの誕生日がわかり、同い年であることから問題無しと考える

---

### S-02｜HPの `since 2023.06` 表記がオープン年と混在して混乱を招く

**重大度：低〜中**

- 店のオープンは `2019年8月26日`。ブラウザのラーメン店HPには `since 2023.06` と表示されている
- これは「リニューアル年」を指すと思われるが、パスコードのヒントとなる「オープン日（08/26）」と混在することで、プレイヤーが「2023年の何かがパスコードに関係する？」と誤解する可能性がある
- **改善案：** HPには元のオープン年 `since 2019.08.26` を残し、リニューアルの事実はSNSの周年記念ポストからのみ読み取れるようにする

→ ブラウザのラーメン店HPからsince 表記を消せば良い
---

### S-03｜太郎が「相方に内緒」なのにLIMEで取材先を話している矛盾

**重大度：低〜中**

- 太郎の最後のSNS投稿：「久々に『あの店』への聖地巡礼。**相方には内緒**にしてサプライズしようかな笑」
- しかしLIMEのチャットログには2024年11月8日（取材日の数日前）に「今週は前から気になってたあの店取材しようと思ってる」「〇ンニクのとこ？」という会話がある
- 「内緒」という言葉と、LIMEで取材先を話していることが矛盾する
- **改善案：** SNSのポストを「（取材内容はまだ内緒だよ〜）笑」に変更するか、LIMEの会話から取材先の具体名を削除する

→ 改善案を採用したい
---

### S-04｜ニュース記事が核心を直接的に開示しすぎている

**重大度：低**（ゲーム性の観点から）

- ニュース記事のハイライトボックスに「**近隣の飲食業者に転売されていた可能性**を含めて調査」と明記されている
- これが最初のブラウザ閲覧時点で表示されるため、謎解きの核心がロック解除直後に与えられてしまう
- プレイヤーが「気づく」体験ではなく「読まされる」体験になりがち
- **改善案：** ハイライト部分を「周辺の複数施設についても任意の調査が行われた模様」程度に留め、「飲食業者への転売」は `revealTrue()` ギミックで初めて明かされるようにする

→ 改善案を採用したい
---

### S-05｜バッドエンドが1ミス即終了で難易度が高すぎる

**重大度：中**

- 謎解きクイズで1問でも誤答すると即座に `startBadEnd()` が呼ばれてゲームオーバー
- バッドエンドの演出自体は優秀だが、「1ミス即終了」はフラストレーションが高い
- **改善案：** 問3（住所の特定）のみをバッドエンド分岐とし、問1・問2は間違えたらヒントを出す仕様にするか、合計ミス数が一定数を超えた場合にバッドエンドになるようにする

→ 問題は３問しかないし、そんなに難しい内容ではない。バッドエンドを一度見るのも楽しいと思うので現行仕様で行く

---

### S-06｜エピローグの日時・視聴者数の描写が薄い

**重大度：低**

- 「無事保護されました」とあるが、いつ救出されたかの描写がない
- 視聴者数「48,291人」は、チャンネル登録者28万人の配信者の「緊急生放送」にしては少なすぎる印象
- **改善案：** 「〇日後のアーカイブ」など具体的な日付を入れる。視聴者数を10〜50万程度に調整する

→ 改善案を採用したい。あるいはもっと面白いエピローグや演出を作成したほうがいい。

## 🖥️ プログラム・UIの問題点

### P-01｜SNSアプリが開いた瞬間に `news_found` ステップへ進んでしまう

**重大度：高**

`sns.html` のスクリプトで、ページ読み込み直後に以下が実行される：

```js
post('SET_FLAG', { key: 'snsVisited', value: true });
post('ADVANCE_STEP', { step: 'news_found' });  // ← これが問題
```

- SNSを開いただけで `news_found` になり、ニュースを読む前にギャラリーのバッジが点灯する
- ブラウザのニュースページを開いた時とも二重に呼ばれている
- **改善案：** SNS読み込み時は `snsVisited` フラグのみ立て、`news_found` へのステップ進行はブラウザのニュースページを開いた時のみに限定する

---

### P-02｜「自分のスマホ」のSNSアプリが太郎のSNSと同一で意味不明

**重大度：中**

- `player-phone` 内の「SNS」アイコンをタップすると `apps/sns.html?preview=1`（太郎のSNS）が開く
- 「自分のスマホ」でなぜ太郎のSNSが見れるのか説明がつかない
- **改善案：** 「自分のスマホのブラウザで太郎のSNSを検索している」という文脈を明示するか、アイコン自体を削除してシンプルにする

→ 自分のスマホのアイコンではなく、ブラウザの検索アイコンタップで太郎のSNSが見れるようにする

---

### P-03｜`renderNextPlayerMessage` のインデックス管理にバグがある

**重大度：高**

`PLAYER_CHAT_SCENARIO` の実際の配列構造：

- インデックス 0〜5：初期会話（6通）
- インデックス 6〜9：ロック解除後（4通）
- インデックス 10〜12：ニュース発見後（3通）
- インデックス 13〜16：グッドエンド充電切れ（4通）
- インデックス 17〜19：バッドエンド充電切れ（3通）= 合計20件

`renderNextPlayerMessage` の条件分岐に以下のバグがある：

- `playerChatIndex === 17` で `triggerClear` を呼んでいる → バグ。17はバッドエンドメッセージの先頭。グッドエンド最後メッセージ表示後のインデックスは `16` になるはず
- `playerChatIndex === 21` でバッドエンドへ → 配列は20件（インデックス19まで）なので21は存在しない

- **改善案：** 終了条件をインデックスのハードコードではなく、シナリオ種別（`'good'`/`'bad'`）で管理し直す

→ 改善案を採用
---

### P-04｜リロード時のチャット復元インデックスが誤り

**重大度：中**

```js
if (gameState.currentStep === 'unlocked') {
    playerChatIndex = 10; // ← 誤り（正しくは10）
}
```

- `unlocked` ステップではインデックス6〜9（ロック解除後の4通）を表示すべきところ、`10` を設定してしまっているためニュース発見後の会話まで先に描画される
- **改善案：** `unlocked` → index 10（表示は0〜9）、`news_found` → index 13（表示は0〜12）と正しく対応させる

→ 改善案を採用

---

### P-05｜ホームバー（home-bar）のUIが分かりにくい

**重大度：中**（ユーザー自身も修正希望.md に記載済み）

- モバイルモードのホームバーのデザインが目立ちにくい
- ホームバーとアプリ本体の境界が不明瞭で、誤タップでアプリが閉じることがある
- **改善案：** ホームバーの高さを増やし、iOS風の丸みのあるホームボタンアイコンを配置。テキストを削除してアイコンのみにする

→ 改善案を採用したい

---

### P-06｜ブラウザの「秘伝の骨」クリックが分かりにくく、競合状態リスクもある

**重大度：中**（ユーザー自身も修正希望.md に記載済み）

- `news_found` ステップ以降でないとクリックしても何も起きない（初見では壊れていると誤解される）
- `GET_STATE` の返答が届く前に描画が完了してしまう競合状態のリスクがある
- **改善案：** `news_found` 前は「秘伝の■■」のように伏せておき、ニュースを読んだ後に文字が出現 → クリック可能になる演出にする

→ ステップの前ではクリックできないようにしておくだけで問題ない

---

### P-07｜ギャラリー：厨房写真を開いただけで `location_spec` に進んでしまう

**重大度：中**

- `openPhoto('kitchen')` 呼び出し時に即座に `ADVANCE_STEP: location_spec` が送信される
- 写真を開いた瞬間に「LIMEに新着メッセージ」通知が鳴るが、プレイヤーはまだクリューモーダルを開いていない
- **改善案：** `ADVANCE_STEP: location_spec` の送信を `confirmLocation()` ボタンを押した時点に移す

→ 厨房写真のくだりはそもそも再考が必要。写真を開いただけなのにいちいちヒントが詳しすぎる。
---

### P-08｜バッドエンドのメッセージ送信者表示が不統一

**重大度：低**

- `startBadEnd()` 内の店主メッセージで `msg-time` の欄に「〇ンニク〇んこつラーメン」という店名を入れている（時刻表示欄に店名が入っているのは設計ミス）
- **改善案：** `msg-time` には時刻を入れ、送信者名は `msg-ava` のtitle属性等で補足する形式に統一する

→ 店名表示欄は不要。時刻のみ表示する

---

### P-09｜「自分のスマホ」の電池残量が常に78%で固定

**重大度：低**

- `player-phone-bar` に `🔋 78%` とハードコードされており変化しない
- **改善案：** 固定値でも許容できるが、`localStorage` に保存して初回表示時に設定するか、ゲーム進行に応じて微妙に減少する演出を追加する

→ 改善案を採用

---

### P-10｜PCとモバイルのリサイズ時にUI状態が混在する

**重大度：低〜中**

- `isMobile()` は `window.innerWidth <= 768` で判定しているが、リサイズイベントへの対応がない
- PC幅でプレイ中に縮小するとPC用ウィンドウとモバイルオーバーレイが混在する可能性がある
- **改善案：** ゲーム開始時の画面幅でモードを固定し、以降はリサイズを無視する方針を徹底する

→ 改善案を採用

---

### P-11｜SNSの周年記念写真が遅い環境でプレースホルダーのままになりうる

**重大度：低**（演出への影響大）

- `sns.html` は画像を `new Image()` で非同期読み込みするため、通信が遅い場合に「📷」プレースホルダーが残ることがある
- 周年記念写真はストーリーの核心的なビジュアルヒントであり、表示されないと謎が解けない
- **改善案：** 画像に `loading="eager"` を設定し、スクリプト先頭でプリロードする

→ 改善案を採用

---

### P-12｜自分のSNSオーバーレイのヘッダー文言が唐突

**重大度：低**

- `player-sns-overlay` のヘッダーに「タベアルキ太郎のSNS」とあるだけで、なぜ自分のスマホで見れるのかが説明されない
- **改善案：** 「📱 太郎のSNS（公開タイムライン）を検索中」などの文言に変更する

→ 改善案を採用

---

### P-13｜`shutdown-overlay` のフェードインがiOS Safariで動作しない可能性がある

**重大度：低**

- `display:flex` 後 50ms で `opacity:1` を設定しているが、ブラウザの再描画タイミングが保証されない
- iOS Safari では `display` と `opacity` のトランジションが同一フレームで処理されてしまいアニメーションが起きないことがある
- **改善案：** `requestAnimationFrame` を使って確実に再描画後に `opacity` を変更する

→ 改善案を採用
